Credential Flow Options

Which deployment model feels right for enterprise onboarding and ingester credentials?

A. Control Plane Stores Secret
Admin UI
Control plane keeps GitHub App private key
↓
Writes secret into cluster for ingester
↓
Ingester uses it at runtime

Full control-plane custody

Easiest admin story, but the closed product permanently holds the sensitive credential.

Pros

  • Smooth UX
  • Central rotation

Cons

  • Weakest trust posture
  • Harder DOD review
B. Customer Secret, Control Plane Metadata
Admin UI
Admin creates secret in customer Secret Manager / K8s
↓
Control plane stores reference + validates config
↓
Ingester reads referenced secret at runtime

Reference, don’t retain

Control plane owns onboarding workflow, but the actual credential stays inside the customer boundary.

Pros

  • Best trust balance
  • Minimal ingester redesign

Cons

  • More admin setup
  • Needs secret-ref UX
C. Control Plane Brokers Runtime Access
Admin UI
Control plane authenticates and brokers GitHub access
↓
Ingester requests short-lived token / proxy session
↓
Repo sync depends on control-plane availability

Brokered access

Most dynamic and centralized, but it pulls the enterprise control plane into the hot path of indexing.

Pros

  • Central policy enforcement
  • No static secret in ingester

Cons

  • Largest redesign
  • New runtime dependency

Recommendation

B gives you enterprise onboarding value without turning the control plane into the long-term holder of repo secrets or forcing the ingester to depend on a remote broker.
