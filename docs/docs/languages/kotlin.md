# Kotlin Parser

This page tracks the checked-in Go parser contract for this branch.
Canonical implementation: `go/internal/parser/registry.go` plus the entrypoint and tests listed below.

## Parser Contract
- Language: `kotlin`
- Family: `language`
- Parser: `DefaultEngine (kotlin)`
- Entrypoint: `go/internal/parser/kotlin_language.go`
- Fixture repo: `tests/fixtures/ecosystems/kotlin_comprehensive/`
- Unit test suite: `go/internal/parser/engine_managed_oo_test.go`
- Integration validation: compose-backed fixture verification (see `../reference/local-testing.md`)

## Known Limitations
- Kotlin interfaces are separately bucketed in the parser, and interface methods now carry `class_context` so interface-typed receiver calls resolve through the normal reducer/query path
- Constructor-injected property receivers are now covered; same-file function-return aliases are now covered too, including re-aliased local chains such as `val provider = createService(); val active = provider; active.info()`, explicit cast assignments such as `val service = any as Service` and direct cast-expression receivers such as `(any as Service).info()` now keep inferred receiver types too, direct object receivers such as `AppConfig.isProduction()` and companion-object receivers such as `Person.create()` are now covered, branch-local smart-cast receivers such as `if (any is Service) any.info()` and `when (value) { is Factory -> value.createService().info() }` now keep inferred receiver types too, safe-call alias assignments now keep inferred receiver types too, receiver-preserving scope-function assignments such as `val service = createService().apply { }` now keep inferred receiver types too, delegated lazy property receivers now keep inferred receiver types too, receiver chains can now carry same-file function-return hops through dotted call paths, chained assignments, and parenthesized call sites, and cross-file package-aware multi-hop chains now resolve across split definition files in the same package. Package-aware sibling lookups now cover both alias-based and direct receiver chains across package-matching sibling directories, the current directory, its parent, its grandparent, and deeper package-root trees, including parenthesized cross-file chains when conflicting packages define same-name helpers, but broader receiver inference beyond explicit typed declarations, explicit cast assignments, direct cast-expression receivers, smart-cast branches, property chains, dotted property aliases, safe-call receiver chains, safe-call alias assignments, receiver-preserving scope-function assignments, delegated lazy property receivers, and larger cross-file/package flow is still partial
